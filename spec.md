# 냉장고를 부탁해 : 자취생편 — 개발 명세서

## 1. 프로젝트 개요

### 1.1 목적

사용자가 현재 냉장고에 보유한 식재료를 등록하고, 보유 재료로 만들 수 있는 레시피를 우선 추천하는 웹 서비스다. 부족한 재료는 별도로 표시하여 장보기와 요리 의사결정을 돕는다.

### 1.2 문제 정의

- 자취생은 보유 식재료를 잊어버리거나 유통기한 전에 활용하지 못한다.
- 레시피를 찾을 때 자신이 가진 재료와 필요한 재료를 일일이 대조해야 한다.
- 결과적으로 식재료 낭비와 음식물 쓰레기가 증가한다.

### 1.3 핵심 사용자 흐름

1. 사용자가 냉장고에 있는 식재료를 검색하거나 목록에서 선택한다.
2. 선택한 재료를 기준으로 레시피를 조회한다.
3. 보유 재료가 많은 순서로 추천 결과를 확인한다.
4. 부족한 재료와 조리 정보를 확인한다.
5. 필요한 레시피를 즐겨찾기에 저장한다.

### 1.4 기본 가정 및 기술 방향

- MVP는 반응형 SPA를 기준으로 한다.
- 레시피 원본은 저장소의 JSON 파일(`public/data/recipes.json`)로 관리한다.
- 별도 백엔드 없이 레시피 JSON을 읽고, 재료 목록과 즐겨찾기는 브라우저 `localStorage`에 저장한다.
- 이후 API 서버를 추가할 수 있도록 데이터 모델과 조회 인터페이스를 분리한다.
- 화면 문구와 데이터의 기본 언어는 한국어로 한다.

## 2. 구현 범위 (In Scope)

- 냉장고 재료 추가: 추천 목록 선택, 검색, 직접 입력
- 재료 중복 및 빈 값 검증
- 등록 재료 조회 및 삭제
- 등록 재료 중 레시피 검색에 사용할 재료 선택/해제
- 선택 재료 기반 레시피 추천
- 보유 재료 충족률 기준 추천 결과 정렬
- 레시피별 보유 재료, 부족 재료, 조리 시간, 난이도, 인분 표시
- 레시피 상세 조회
- 즐겨찾기 추가/해제 및 목록 조회
- 새로고침 후 냉장고 재료와 즐겨찾기 유지
- 로딩, 빈 상태, 오류 상태 처리
- 모바일/데스크톱 반응형 UI

## 3. 제외 범위 (Out of Scope)

- 회원가입, 로그인, 서버 기반 사용자 계정

- 실제 재료 구매, 결제, 배달 및 장보기 주문
- 실시간 재고, 가격, 영양 정보, 알레르기 정보의 외부 연동
- 유통기한 자동 인식 및 알림
- 이미지 인식으로 식재료를 등록하는 기능
- 사용자가 레시피를 작성·수정·공유하는 기능
- 자연어 기반 AI 추천 및 개인화 학습
- 레시피 JSON 관리자 편집 기능
- 서버 DB 및 실시간 추천 랭킹

## 4. 기능 요구사항

### FR-01. 재료 목록 조회

- 초기 진입 시 기본 재료 목록을 표시한다.
- 기본 재료는 별도 `ingredients` 데이터 또는 레시피 JSON에서 중복 제거해 생성할 수 있다.
- 등록 재료는 “보유 재료”, 검색 결과는 “추가 가능한 재료”로 구분한다.

### FR-02. 재료 추가

- 사용자는 재료 이름을 검색하거나 직접 입력해 추가할 수 있다.
- 입력값은 앞뒤 공백을 제거한다. 빈 문자열은 추가할 수 없다.
- 동일 이름의 재료는 중복 추가할 수 없다. 비교 시 한글/영문 대소문자를 구분하지 않는다.
- 추가 성공 시 냉장고 목록에 즉시 반영하고 선택 상태는 기본적으로 선택으로 설정한다.
- 추가 실패 시 원인을 알 수 있는 안내 문구를 표시한다.

### FR-03. 재료 삭제

- 보유 재료마다 삭제 버튼을 제공한다.
- 삭제된 재료는 선택 재료에서도 자동 해제한다.
- 삭제 후 추천 결과를 즉시 다시 계산한다.
- 삭제 결과를 `localStorage`에 즉시 반영한다.

### FR-04. 재료 선택/해제

- 보유 재료별 체크박스 또는 토글을 제공한다.
- “전체 선택”, “전체 해제”를 제공한다.
- 선택 재료가 0개이면 추천 실행 전 안내 문구를 표시한다.
- 선택 상태는 새로고침 후에도 유지한다.

### FR-05. 레시피 추천

- 선택 재료를 기준으로 레시피의 일치 재료 수와 부족 재료 수를 계산한다.
- 최소 한 개 이상의 재료가 일치하는 레시피만 기본 결과에 표시한다.
- 추천순은 일치율, 부족 재료 수, 조리 시간, 레시피명 순으로 정렬한다.
- “추천순”, “조리 시간순” 정렬을 제공한다.
- 카드에는 레시피명, 대표 이미지, 일치율, 보유 재료 수, 부족 재료 수, 난이도를 표시한다.

### FR-06. 부족 재료 표시

- 상세 및 카드에서 필요한 재료를 보유/부족으로 구분한다.
- 부족 재료에는 재료명과 필요 수량을 표시한다.
- 선택되지 않은 보유 재료는 추천 계산상 보유 재료로 인정하지 않는다.

### FR-07. 레시피 상세 조회

- 레시피 카드 클릭 또는 상세 버튼으로 상세 화면/모달을 연다.
- 상세에는 레시피명, 설명, 이미지, 조리 시간, 난이도, 인분, 전체 재료, 조리 순서, 즐겨찾기 상태를 표시한다.
- 존재하지 않는 ID는 “레시피를 찾을 수 없습니다” 상태를 표시한다.

### FR-08. 즐겨찾기

- 카드와 상세 화면에서 즐겨찾기 추가/해제를 할 수 있다.
- 즐겨찾기 목록은 별도 메뉴 또는 페이지에서 조회한다.
- 즐겨찾기 상태는 `localStorage`에 저장하고 새로고침 후 유지한다.
- JSON 로딩 실패 시 기존 즐겨찾기 ID를 임의로 삭제하지 않는다.

### FR-09. 상태 및 접근성

- JSON 로딩 중에는 로딩 UI를 표시한다.
- 로딩 실패 시 재시도 버튼과 오류 메시지를 표시한다.
- 결과가 없을 때 원인별 빈 상태를 표시한다: 선택 재료 없음, 일치 레시피 없음, 즐겨찾기 없음.
- 버튼과 입력 요소는 키보드로 조작 가능해야 한다.
- 아이콘만 있는 버튼에는 `aria-label`을 제공한다.
- 색상만으로 보유/부족 상태를 구분하지 않는다.

## 5. 페이지별 요구사항

### 5.1 홈/냉장고 관리 페이지 (`/`)

구성:

- 서비스명과 간단한 설명
- 재료 검색 입력창
- 추천 재료 칩/버튼 목록
- 직접 입력 및 추가 버튼
- 보유 재료 목록
- 각 재료의 선택 토글과 삭제 버튼
- 전체 선택/해제 버튼
- “레시피 추천하기” CTA
- 즐겨찾기 페이지 이동 네비게이션

동작:

- 재료 추가·삭제·선택 상태를 즉시 반영한다.
- CTA 클릭 시 선택 재료를 전역 상태 또는 `localStorage`에 저장하고 `/recipes`로 이동한다.
- 보유 재료가 없으면 안내 문구와 예시 재료를 제공한다.

### 5.2 추천 결과 페이지 (`/recipes`)

구성:

- 현재 선택한 재료 요약
- 결과 개수
- 정렬 셀렉트: 추천순/조리 시간순
- 필터: 전체, 재료를 모두 보유, 부족 재료 3개 이하
- 레시피 카드 그리드
- 홈으로 돌아가기 버튼

동작:

- 새로고침 또는 직접 URL 접근 시 `localStorage`의 선택 재료를 사용한다.
- 선택 재료가 없으면 안내하고 홈으로 돌아갈 수 있게 한다.
- 카드 클릭 시 `/recipes/:recipeId`로 이동한다.

### 5.3 레시피 상세 페이지 (`/recipes/:recipeId`)

구성:

- 뒤로가기, 제목, 설명, 이미지
- 조리 시간, 난이도, 인분
- 보유/부족 재료 섹션
- 조리 순서
- 즐겨찾기 토글

동작:

- URL의 `recipeId`로 JSON에서 레시피를 조회한다.
- 홈에서 선택한 재료가 있으면 보유/부족 상태를 계산한다.
- 선택 재료가 없어도 전체 재료 정보와 즐겨찾기를 조회할 수 있다.

### 5.4 즐겨찾기 페이지 (`/favorites`)

- 저장된 ID를 레시피 데이터와 매칭해 카드 목록을 표시한다.
- 카드 클릭 시 상세 페이지로 이동한다.
- 해제 버튼을 누르면 즉시 목록에서 제거한다.
- 즐겨찾기가 없으면 빈 상태 안내를 표시한다.

## 6. 데이터 모델

### 6.1 레시피 JSON

파일 경로: `public/data/recipes.json`

```json
[
  {
    "id": "recipe-001",
    "title": "간장 계란밥",
    "description": "간단한 재료로 만드는 한 끼 식사",
    "imageUrl": "/images/recipes/soy-egg-rice.jpg",
    "servings": 1,
    "cookingTimeMinutes": 10,
    "difficulty": "easy",
    "ingredients": [
      { "name": "밥", "amount": "1공기", "required": true },
      { "name": "계란", "amount": "1개", "required": true },
      { "name": "간장", "amount": "1큰술", "required": true },
      { "name": "참기름", "amount": "약간", "required": false }
    ],
    "steps": [
      "계란을 프라이한다.",
      "밥 위에 계란을 올리고 간장과 참기름을 곁들인다."
    ],
    "tags": ["한그릇", "자취요리", "초간단"]
  }
]
```

필드 규칙:

| 필드 | 타입 | 필수 | 규칙 |
|---|---|---:|---|
| `id` | string | O | 전체 JSON에서 유일한 slug |
| `title` | string | O | 화면에 표시할 레시피명 |
| `description` | string | O | 한 줄 설명 |
| `imageUrl` | string/null | X | 없으면 기본 이미지 사용 |
| `servings` | number | O | 1 이상의 정수 |
| `cookingTimeMinutes` | number | O | 0보다 큰 정수 |
| `difficulty` | `easy\|medium\|hard` | O | 쉬움/보통/어려움으로 표시 |
| `ingredients` | `Ingredient[]` | O | 1개 이상 |
| `steps` | string[] | O | 순서가 보장된 조리 단계 |
| `tags` | string[] | X | 필터/검색용 |

### 6.2 브라우저 저장 데이터

```json
{
  "fridge-ingredients": ["밥", "계란", "대파"],
  "selected-ingredients": ["밥", "계란"],
  "favorite-recipe-ids": ["recipe-001", "recipe-004"]
}
```

- 실제 구현에서는 각 키를 개별 `localStorage` 항목으로 저장한다.
- 재료명은 `trim()` 후 연속 공백을 하나로 치환하고 소문자로 변환해 비교한다.
- 표시용 원본 이름은 최초 등록된 표기를 유지한다.
- 저장 데이터가 파싱에 실패하거나 배열이 아니면 해당 항목을 빈 배열로 복구한다.

### 6.3 런타임 추천 결과

```ts
type RecipeMatch = {
  recipe: Recipe;
  matchedIngredients: Ingredient[];
  missingIngredients: Ingredient[];
  matchedCount: number;
  missingCount: number;
  matchRate: number; // 0~1
  allRequiredAvailable: boolean;
};
```

## 7. 추천 알고리즘

### 7.1 재료 일치 판정

1. 사용자 선택 재료와 레시피 재료명을 동일한 정규화 함수로 변환한다.
2. 레시피 재료마다 선택 재료 집합에 포함되는지 확인한다.
3. 포함되면 `matchedIngredients`, 아니면 `missingIngredients`에 넣는다.
4. MVP에서는 `required: false` 재료도 일치율 계산에 포함한다. 운영 데이터 전체에 같은 규칙을 적용한다.

### 7.2 점수 및 정렬

```text
matchRate = matchedCount / ingredients.length
```

추천순 정렬 우선순위:

1. `matchRate` 내림차순
2. `missingCount` 오름차순
3. `cookingTimeMinutes` 오름차순
4. `title` 가나다순

표시 대상은 `matchedCount >= 1`인 레시피다. “재료를 모두 보유”는 `missingCount === 0`, “부족 재료 3개 이하”는 `missingCount <= 3`으로 필터링한다.

예시: 선택 재료가 `[밥, 계란, 대파]`, 레시피 재료가 `[밥, 계란, 간장, 참기름]`이면 보유 2개, 부족 2개, 일치율 50%다.

## 8. API 인터페이스 (선택 사항)

MVP는 정적 JSON을 직접 조회하므로 백엔드 API가 필수는 아니다. API 서버로 전환할 때 아래 인터페이스를 유지한다.

### 8.1 레시피 목록 조회

```http
GET /api/recipes?ingredients=밥,계란&sort=recommended&maxMissing=3
```

응답 `200 OK`:

```json
{
  "items": [
    {
      "id": "recipe-001",
      "title": "간장 계란밥",
      "matchedCount": 2,
      "missingCount": 1,
      "matchRate": 0.67,
      "matchedIngredients": ["밥", "계란"],
      "missingIngredients": ["간장"]
    }
  ],
  "total": 1
}
```

### 8.2 레시피 상세 조회

```http
GET /api/recipes/:recipeId
```

- 성공: `200` + `Recipe` 객체
- 존재하지 않음: `404` + `{ "code": "RECIPE_NOT_FOUND", "message": "..." }`

### 8.3 즐겨찾기

MVP에서는 API 없이 `localStorage`를 사용한다. 서버 저장으로 확장할 때 다음 계약을 사용한다.

```http
GET    /api/favorites
POST   /api/favorites/:recipeId
DELETE /api/favorites/:recipeId
```

- 성공 응답: `200` 또는 `201`
- 잘못된 recipe ID: `404`
- 인증이 필요한 환경에서 인증 없음: `401`

## 9. 비기능 요구사항

- 최신 일반 브라우저(Chrome, Edge, Safari, Firefox)에서 동작한다.
- 일반적인 개발 환경에서 레시피 JSON 요청 후 2초 이내에 사용할 수 있어야 한다.
- 레시피가 500개까지 늘어도 클라이언트 필터링과 정렬이 눈에 띄게 지연되지 않아야 한다.
- 모든 상태 변경 후 화면과 `localStorage`가 동일한 상태를 유지해야 한다.
- 이미지 로딩 실패 시 레이아웃이 깨지지 않고 대체 이미지를 표시한다.

## 10. Acceptance Criteria

### AC-01 재료 관리

- [ ] 목록에서 재료를 추가하면 보유 재료 영역에 즉시 나타난다.
- [ ] 같은 재료를 두 번 추가해도 한 번만 표시된다.
- [ ] 빈 문자열 또는 공백만 입력하면 추가되지 않는다.
- [ ] 삭제한 재료는 보유 목록과 선택 목록에서 모두 제거된다.
- [ ] 새로고침해도 보유 재료가 유지된다.

### AC-02 선택 및 추천

- [ ] 재료별 선택/해제가 가능하다.
- [ ] 선택 재료가 없을 때 안내 상태가 나타난다.
- [ ] 하나 이상의 선택 재료와 일치하는 레시피만 결과에 나타난다.
- [ ] 각 결과에서 일치 수, 부족 수, 일치율을 확인할 수 있다.
- [ ] 추천순은 일치율이 높은 레시피를 먼저 표시한다.
- [ ] 조리 시간순은 조리 시간이 짧은 순으로 표시한다.

### AC-03 상세 및 부족 재료

- [ ] 카드를 클릭하면 해당 레시피 상세를 확인할 수 있다.
- [ ] 상세에 재료, 수량, 조리 시간, 난이도, 인분, 조리 순서가 표시된다.
- [ ] 보유 재료와 부족 재료가 시각적으로 구분된다.
- [ ] 잘못된 ID에서는 오류 상태가 표시되고 앱이 중단되지 않는다.

### AC-04 즐겨찾기

- [ ] 카드 또는 상세에서 즐겨찾기를 추가·해제할 수 있다.
- [ ] 즐겨찾기 페이지에 저장한 레시피만 나타난다.
- [ ] 새로고침 후에도 즐겨찾기 상태가 유지된다.
- [ ] 모두 해제하면 빈 상태 안내가 나타난다.

### AC-05 예외 및 접근성

- [ ] `recipes.json` 요청 실패 시 오류 메시지와 재시도 버튼이 나타난다.
- [ ] 레시피가 없을 때 빈 상태 안내가 나타난다.
- [ ] 핵심 버튼과 입력 요소를 키보드만으로 사용할 수 있다.
- [ ] 아이콘 버튼에 접근 가능한 이름이 있다.
- [ ] 모바일 화면에서 가로 스크롤 없이 핵심 기능을 사용할 수 있다.

## 12. 향후 확장 사항

- 로그인 및 사용자별 냉장고 데이터 서버 저장
- 유통기한 입력, 임박 재료 우선 추천, 알림
- 재료 카테고리·수량·단위 관리
- 영양 성분, 알레르기 및 식단 목표 기반 필터
- 부족 재료 장보기 목록 자동 생성
- 레시피 리뷰, 평점, 사용자 레시피 등록
- 재료 대체재 추천
- 가격·재고·온라인 장보기 API 연동
- 사용자의 조회·즐겨찾기 이력을 활용한 개인화 추천
- PWA 및 오프라인 캐시 지원
