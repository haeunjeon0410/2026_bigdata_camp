# AI 에이전트 기능 정리

## 1. 구현 여부

본 프로젝트에는 영수증 이미지를 분석하는 AI 기능이 구현되어 있습니다.

다만 사용자의 요청을 스스로 계획하고 여러 도구를 반복 실행하는 자율형 에이전트라기보다는, OpenAI Vision API를 호출해 영수증 상품을 추출하는 AI 분석 기능입니다.

## 2. 기능 개요

- 기능명: 영수증 이미지 분석 및 냉장고 재료 등록
- 입력: 사용자가 업로드한 영수증 이미지
- AI 모델: OpenAI `gpt-4o-mini`
- 처리 위치: Supabase Edge Function
- 결과: 상품명, 일반 재료명, 수량을 JSON으로 반환
- 후속 처리: 사용자가 분석 결과를 확인하고 선택한 재료를 냉장고 목록에 추가

## 3. 동작 흐름

1. 사용자가 영수증 이미지를 업로드합니다.
2. 프론트엔드가 Supabase `analyze-receipt` Edge Function을 호출합니다.
3. Edge Function이 이미지를 Base64로 변환해 OpenAI Vision API에 전달합니다.
4. AI가 영수증의 식품 상품을 추출하고 `items` JSON으로 응답합니다.
5. 프론트엔드가 결과를 정규화하고 중복 상품을 제거합니다.
6. 사용자가 추가할 상품을 선택하면 냉장고 재료 목록에 등록합니다.

## 4. 관련 파일

- `supabase/functions/analyze-receipt/index.ts`: OpenAI Vision API 호출 및 결과 JSON 반환
- `js/supabase.js`: Supabase Edge Function 호출 어댑터
- `js/receipt.js`: 영수증 업로드, 분석 상태 표시, 결과 선택 및 등록 처리
- `js/ingredient.js`: 분석된 재료의 냉장고 등록 처리

## 5. 보안 및 오류 처리

- OpenAI API 키는 클라이언트에 노출하지 않고 Supabase Secret `OPENAI_API_KEY`로 관리합니다.
- 영수증 이미지 크기는 최대 10MB로 제한합니다.
- 이미지에 보이지 않는 상품을 추측하지 않도록 분석 프롬프트에 제한을 둡니다.
- 비식품 항목, 할인, 배송비, 결제금액 등은 결과에서 제외하도록 요청합니다.
- OpenAI 오류, JSON 파싱 오류, 빈 분석 결과에 대해 사용자 오류 상태를 표시합니다.

## 7. 실행에 필요한 설정

Supabase Edge Function 환경 변수에 다음 Secret을 설정해야 실제 AI 분석이 동작합니다.

```text
OPENAI_API_KEY=실제 OpenAI API 키
```

Supabase 연결 정보가 없거나 API Secret이 설정되지 않은 환경에서는 실제 분석을 수행할 수 없습니다.

## 8. 기능 추가 필요 여부

현재 요구사항이 “AI 에이전트 기능 구현 여부와 작업 내역 제출”이라면 추가 기능 없이도 제출 가능한 상태입니다.

현재 구현은 AI 기반 영수증 분석 기능이며, 자율형 에이전트임을 명확히 요구받는 경우에만 다음과 같은 확장을 검토할 수 있습니다.

- 분석 결과에 따라 자동으로 재료를 분류하고 저장 위치를 결정하는 단계
- 보유 재료를 바탕으로 레시피를 선택하고 추천 이유를 생성하는 단계
- 여러 단계의 도구 호출과 실패 재시도, 사용자 확인을 포함하는 에이전트 흐름

## 9. 평가 기준별 정리

### 해결 트랙

**생활 자동화·멀티모달 AI 트랙**에 해당합니다.

사용자가 직접 영수증의 상품명을 하나씩 입력하는 불편을 해결하기 위해, 이미지 입력을 AI가 읽고 냉장고 재료 데이터로 연결합니다. 즉, 단순한 질의응답보다 실제 생활 데이터를 구조화해 다음 행동으로 이어지게 하는 문제 해결형 기능입니다.

### 사용 도구

- OpenAI Vision API: 영수증 이미지의 상품명과 수량 인식
- Supabase Edge Function: API 키 보호, 이미지 분석 요청의 서버 처리
- Supabase Functions SDK: 프론트엔드와 분석 함수 연결
- 기존 재료 데이터 및 매칭 로직: AI 결과를 서비스 내부 재료 체계에 연결

### RAG 적용 여부

현재 영수증 분석 단계에는 별도의 벡터 DB나 문서 검색 기반 RAG를 적용하지 않았습니다.

그 이유는 영수증 분석의 핵심이 외부 지식 검색이 아니라 이미지에 실제로 보이는 상품명을 정확히 읽는 것이기 때문입니다. 이 단계에 임의의 레시피 문서나 외부 상품 지식을 검색해 넣으면 이미지에 없는 상품을 추측할 위험이 커집니다.

대신 다음과 같은 제한된 지식 연결을 사용합니다.

- 서비스 내부 재료 데이터와 상품명 매칭
- 매칭되지 않은 분석 결과의 런타임 재료 등록
- 기존 레시피 데이터와 보유 재료를 이용한 추천 필터링

향후 RAG를 확장한다면, 분석된 재료를 기준으로 내부 레시피·보관법·대체 재료 문서를 검색해 추천 근거를 제공하는 방식이 적합합니다. 영수증 OCR 자체에는 RAG를 강제로 적용하지 않는 것이 더 정확한 설계입니다.

### 응답 설계

AI 응답은 자유로운 자연어가 아니라 다음 JSON 구조로 제한했습니다.

```json
{
  "items": [
    {
      "rawName": "영수증에 표시된 상품명",
      "normalizedName": "서비스에서 사용할 일반 재료명",
      "quantity": "수량 또는 중량"
    }
  ]
}
```

이 방식을 선택한 이유는 다음과 같습니다.

1. 프론트엔드가 결과를 안정적으로 파싱할 수 있습니다.
2. 영수증 원문과 서비스용 재료명을 분리해 검증할 수 있습니다.
3. 사용자가 냉장고에 추가할 항목을 직접 확인하고 선택할 수 있습니다.
4. 자유 응답에 포함될 수 있는 설명 문장이나 잘못된 추측을 후속 처리에서 차단할 수 있습니다.

### 왜 이 방식으로 만들었는가

영수증은 이미지이므로 텍스트 입력보다 Vision 모델이 적합합니다. API 키는 브라우저에 노출되면 안 되므로 Supabase Edge Function을 중간 계층으로 사용했습니다. 또한 AI가 냉장고 데이터를 바로 변경하지 않고 분석 결과를 먼저 사용자에게 보여준 뒤 선택하게 했습니다.

이 구조는 다음 원칙을 만족합니다.

- 이미지 인식은 AI에 맡기고, 재료 등록은 기존 서비스 로직이 담당합니다.
- AI의 자유로운 판단 범위를 줄이고 구조화된 결과만 받습니다.
- 잘못 읽은 상품이 자동으로 저장되는 것을 막아 사용자 통제권을 유지합니다.
- 별도 RAG가 필요한 문제와 필요하지 않은 문제를 분리해 불필요한 검색과 환각을 줄입니다.

### 발표용 한 문장

> 사용자의 영수증 이미지를 OpenAI Vision으로 구조화하고, Supabase Edge Function으로 안전하게 처리한 뒤, 서비스 내부 재료 데이터와 연결해 사용자가 검토 후 냉장고에 등록하도록 설계했습니다. 영수증 판독 자체는 외부 지식 검색보다 이미지 근거가 중요하므로 RAG를 적용하지 않았고, 이후 레시피 추천 단계에서 내부 레시피 문서를 검색하는 방향으로 확장할 수 있습니다.
