# 개발 명세서 (spec.md)

## 1. 프로젝트 구조
현재 프로젝트의 폴더 및 파일 구조는 다음과 같습니다.

```text
2026_bigdata_camp/
├── index.html                # 서비스 메인 마크업 템플릿
├── style.css                 # 전체 레이아웃 및 컴포넌트 디자인 스타일시트
├── 1.png                     # 홈 화면 대표 일러스트 리소스
├── README.md                 # 프로젝트 통합 안내 파일
├── docs/                     # 프로젝트 기획 및 명세 문서 디렉토리
│   ├── PRD.md                # 기획 제품 요구사항 문서 (Product Requirements Document)
│   └── spec.md               # 개발 명세서 및 협업 규칙 문서 (본 문서)
└── js/                       # 자바스크립트 모듈 소스코드 디렉토리
    ├── app.js                # 메인 런타임 상태, SPA 라우팅 및 통합 엔트리포인트
    ├── data.js               # 식재료 및 레시피 데이터 모듈
    ├── favorite.js           # 즐겨찾기(하트 찜) 기능 이벤트 모듈
    ├── ingredient.js         # 냉장고 내 식재료 탐색, 선택, 도어 제어 이벤트 모듈
    ├── recipe.js             # 상세 화면 진입, 조리 프로그레스, 별점 평가 제어 이벤트 모듈
    ├── recommend.js          # 레시피 캐러셀 슬라이드, 편의점 레시피 전환 이벤트 모듈
    ├── shopping.js           # 장보기 리스트 체크 토글 모듈
    └── supabase.js           # 향후 Supabase 서버 연동을 위한 클라이언트 브릿지 모듈
```

## 2. 파일별 역할 상세
프로젝트를 구성하는 각 핵심 파일들의 구현 범위와 상세 역할은 아래와 같습니다.

### index.html
- **역할**: 웹 어플리케이션의 유일한 HTML 문서(SPA)
- **상세**:
  - 헤더 영역(네비게이션 바 및 햄버거 메뉴), 메인 뷰포트 영역(`<main id="app">`), 알림용 토스트 컨테이너 레이아웃 정의
  - 폰트(Google Fonts) 및 전체 CSS 연동
  - 모듈 진입점인 `js/app.js`를 로드 (`<script type="module" src="./js/app.js"></script>`)

### style.css
- **역할**: 웹 서비스의 비주얼 테마 및 애니메이션 디자인
- **상세**:
  - HSL 타이포그래피 기반의 아기자기하고 편안한 자취 감성의 컬러 팔레트 구축
  - 냉장고 캐비닛 열기/닫기 스윙(3D Perspective Transform) 스타일 지정
  - 재료 칩 흔들림 효과, 버튼 클릭 튀어오름 효과, 완료 축하 Confetti 파티클 애니메이션 등 상세 제어

### js/app.js
- **역할**: 전역 상태 관리 및 SPA 라우팅을 총괄하는 코어 엔진이자 단일 엔트리포인트
- **상세**:
  - 어플리케이션의 런타임 상태 객체(`state`: 선택된 재료, 라우트 경로, 캐러셀 인덱스, 별점 결과 등) 관리
  - 해시가 필요 없는 함수 호출 기반 라우터(`navigate`) 및 전체 페이지 컴포넌트 렌더링 스위치(`render`) 제공
  - 토스트 팝업 헬퍼, 조리 시작 타이머 제어, 불꽃 축하 효과 바인딩
  - 하단부에서 순환 참조 회피를 위한 비동기 동적 임포트(Dynamic Import)를 통해 개별 하위 이벤트 모듈들을 순차적으로 바인딩 및 실행

### js/ingredient.js
- **역할**: 보유 식재료 선택 및 필터 관리 제어
- **상세**:
  - 냉장고 도어 노크 시 문 열기/닫기 상태 토글 감지
  - 재료 스티커 선택 상태 바구니 동기화 및 튕김 보폭 애니메이션 제어
  - 검색창 필터링 입력값 감지 및 도어 포켓 카테고리 탭 클릭 시의 재료 리스트 실시간 필터 적용

### js/recommend.js
- **역할**: 레시피 추천 및 캐러셀 슬라이더 제어
- **상세**:
  - 추천 목록 좌우 스와이프 버튼 및 도트 페이지 네이션 매핑
  - "마음에 드는게 없어요 😢" 클릭 시 편의점 꿀조합 리스트 <-> 일반 리스트 간의 양방향 전환 제어

### js/recipe.js
- **역할**: 레시피 세부 스텝 조회 및 조리/시식 동작 제어
- **상세**:
  - 추천 카드 클릭 시 해당 아이디의 상세 레시피/영수증 뷰 네비게이션 트리거
  - 요리 도감 내 잠금해제된 발견 카드 클릭 시 상세 이동 제어
  - 요리 완료 후 시식 피드백 별점 저장 및 도감 데이터 누적 반영

### js/favorite.js
- **역할**: 레시피 하트 즐겨찾기 저장 제어
- **상세**:
  - 카드 우상단 및 상세 페이지의 하트(❤️) 버튼 클릭 이벤트 전파 차단 및 선택 토글
  - 즐겨찾기 저장 목록에 즉시 추가/해제하고 완료 토스트 노출

### js/shopping.js
- **역할**: 부족한 재료 자동 분류 및 장보기 체크리스트 관리
- **상세**:
  - 레시피 영수증 뷰포트 내부에서 부족한 품목 클릭 시 장보기 체크 상태(취소선 및 체크박스 상태값) 토글

### js/data.js
- **역할**: 프로젝트의 핵심 원본 데이터 및 유틸 함수 공급
- **상세**:
  - 원본 식재료(Ingredients) 목록, 레시피(Recipes) 목록, 편의점 대체 요리(Alternative Recipes) 목록 정의
  - 특정 재료 및 레시피를 ID 기반으로 탐색하는 조회 헬퍼 API(`findIngredient`, `findRecipe`) 제공

### js/supabase.js
- **역할**: 클라우드 Supabase DB 동기화용 선제적 어댑터 레이어
- **상세**:
  - 향후 Supabase 활성화 시 사용할 클라이언트 초기화 뼈대 제공
  - 로컬 Mock 함수와 Supabase 테이블 통신 분기 처리를 통해 안정적인 마이그레이션 기반 마련

## 3. 역할 분담 및 담당자
각 팀원별 작업 기여 영역은 다음과 같이 엄격하게 구분됩니다.

| 담당자 | 담당 영역 및 작업 파일 |
|---|---|
| **하은 (팀장)** | 프로젝트 관리, 초기 세팅, Git 관리, `index.html`, `style.css`, `js/app.js` (최종 병합) |
| **소민** | 재료 관리 담당, `js/ingredient.js` |
| **지호** | 추천 기능 담당, `js/recommend.js` |
| **형묵** | 레시피 상세, 즐겨찾기 이벤트, 장보기 기능 담당, `js/recipe.js`, `js/favorite.js`, `js/shopping.js` |
| **채린** | 데이터 원본 관리, Supabase 어댑터 연동 담당, `js/data.js`, `js/supabase.js` |

## 4. 수정 금지 규칙
원활한 코드 통합과 충돌 방지를 위해 아래 규칙을 절대 준수합니다.

> [!WARNING]
> **팀원들은 본인 담당 파일이 아닌 아래 핵심 파일을 절대 임의로 수정할 수 없습니다.**
> - `index.html` (하은 담당)
> - `style.css` (하은 담당)
> - `js/app.js` (하은 담당)
>
> 기능 추가나 스타일 수정이 필요한 경우, 반드시 해당 파일 담당자(하은)에게 검토 및 수정을 요청해야 합니다.

## 5. Git 협업 규칙
- **Direct Push 금지**: `main` 브랜치에 직접 코드를 커밋 및 push하는 행위는 차단됩니다.
- **브랜치 활용**: 본인의 이름 또는 담당 기능 이름으로 브랜치를 별도 생성하여 작업합니다 (예: `feature/ingredient-somin`).
- **Pull Request(PR) 활용**: 작업 완료 후 `main` 브랜치 방향으로 PR을 접수하고, 코드 리뷰 및 최종 병합 권한을 가진 팀장의 승인 후 Merge합니다.
- **최신화 유지**: 작업 시작 전 반드시 로컬 `main` 브랜치를 최신 상태로 `git pull` 완료한 뒤 작업을 파생시킵니다.
- **충돌(Conflict) 해결**: Merge 중 충돌 발생 시, 임의로 합치지 말고 즉시 팀장에게 공유하여 함께 해결책을 마련합니다.

## 6. AI 사용 규칙
GitHub Copilot, ChatGPT 등의 AI 코딩 어시스턴트를 프로젝트에 도용하는 경우, 코드 안전성을 위해 다음 제한 규칙을 준수합니다.

> [!CAUTION]
> - **범위 제한**: AI를 활용해 코드를 작성할 때에도 **지정된 담당 파일 내부의 함수로만 범위를 제한**합니다.
> - **임의 파일 생성 금지**: 사전에 지정되지 않은 자바스크립트나 리소스 파일을 새로 추가하지 마십시오.
> - **핵심 자산 수정 불가**: AI 결과물이라 할지라도 `index.html`, `style.css`, `js/app.js` 파일은 수정할 수 없으며, 기존 HTML 마크업 계층 구조 및 CSS Class/ID 명칭을 파괴하거나 변경해서는 안 됩니다.

## 7. 개발 및 품질 규칙
- **주석 준수**: 모든 함수 선언문 상단에는 해당 함수의 목적, 매개변수(params), 반환값(returns)을 명시한 JSDoc 형식 또는 한 줄 설명 주석을 필수 작성합니다.
- **네이밍 룰**: 모든 자바스크립트 변수, 상수 및 함수 이름은 직관적인 **영어 카멜케이스(camelCase)**를 사용합니다.
- **콘솔 청소**: 디버깅 목적으로 활용한 `console.log`, `console.error` 구문은 기능 완료 후 깨끗하게 삭제(혹은 주석 처리)한 상태로 Pull Request를 접수합니다.
- **코드 미니멀리즘**: 사용되지 않는 레거시 함수나 중복 임포트, 죽은 코드(Dead Code)는 프로젝트 내에 방치하지 않고 즉시 지웁니다.
- **자가 테스트**: 본인 브랜치를 Merge 요청하기 전, 로컬 개발 서버 환경에서 자신이 변경한 기능이 전체 레이아웃과 충돌 없이 잘 조화되는지 브라우저에서 최종 테스트를 진행한 뒤 PR을 생성합니다.
